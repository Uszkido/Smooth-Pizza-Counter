02 🔌 How to Configure the Port

The server port is controlled by a single key in config.json, located in the same folder as PizzaClub.exe. Open it with any text editor (Notepad, VS Code, etc.).

Step 1 — Edit config.json

Find the "app_port" key and change the number to your desired port:

config.json{
    "cam_ip":      "192.168.1.16",
    "cam_port":   "554",

    /* ── Change this number to any FREE port ── */
    "app_port":   8040,

    "branch_name": "Old Airport",
    /* ... rest of config ... */
}
⚠️ Pick a port that is not already in use
Common ports to avoid: 80, 443, 3000, 3306 (MySQL), 5432 (PostgreSQL), 8080 (other web apps). Safe choices: 8040, 8050, 8888, 9000, 5000.

Step 2 — Launch the App

There are two ways to open the dashboard after editing the port:

A
Double-click CHANGE_PORT_AND_LAUNCH.bat This reads the port from config.json, starts PizzaClub.exe, waits 5 seconds, then opens your browser directly on the correct URL — e.g. http://127.0.0.1:8040. Use this when the server is already listening on the right port.
B
Double-click PizzaClub.exe directly The tray app reads config.json and opens your browser on the configured URL automatically. This is the standard launch method.
C
Open browser manually Type http://127.0.0.1:<your_port> into any browser after the app starts.

Remote Access (ngrok) with New Port

START_REMOTE_ACCESS.bat automatically reads the port from config.json and passes it to ngrok. No manual edits to the batch file are needed.

💡 Multi-branch deployments
Each machine can have a different "app_port" in its own config.json. This allows running multiple instances on the same Windows machine — one per branch — on different ports.

Important: Compiled EXE vs. Source Code

🔴 The existing EXE has the port compiled inside it

PyInstaller freezes the Python source into the .exe at build time. The current PizzaClub.exe in this zip was compiled before the port-config feature existed, so the server binding inside it is still hardcoded to the old port.

Solution: Run BUILD_NEW_EXE.bat from the project root to recompile. This produces a new dist_FINAL\PizzaVMS\PizzaVMS.exe that fully respects "app_port" at runtime.

03 🎥 Video Performance Improvements

The live camera feed is the most CPU-intensive component of the system. Three targeted tweaks were made to the frame pipeline to reduce latency, increase throughput, and improve visual clarity.

Parameter Before After Effect
Frame throttle interval 30 ms 20 ms Effective cap raised from ~33 fps to ~50 fps
Idle sleep between frames 10 ms 5 ms Lower inter-frame latency, more responsive feel
Resize width (YOLO input) 1280 px 960 px Faster YOLO inference + JPEG encode per frame
JPEG stream quality 75 85 Noticeably sharper image in the browser viewer
💡 Why 960 px instead of 1280 px?
The YOLO model runs at imgsz=480 regardless — it internally downscales the frame. Feeding it a 960px-wide frame rather than 1280px means less pixel data to resize internally, faster memory transfers, and a smaller JPEG to compress and stream. Counting accuracy is completely unaffected.

Additional Tips to Maximise Smoothness

  • Use a 2MP (1080p) IP camera — 4MP streams waste CPU on decode before YOLO sees a single pixel.
  • Connect the camera via wired Ethernet, not Wi-Fi, to avoid RTSP dropout frames.
  • Keep the PC plugged into power — Windows power-saving limits CPU performance on battery.
  • Close other browser tabs showing the live feed if only one viewer is needed.

04 🛡️ Enterprise: Watchdog & Silent Startup

For mission-critical shop floor environments, the system now includes an autonomous Watchdog and a Silent Startup mechanism. This ensures the AI engine stays alive even if it crashes or the PC restarts accidentally.

1. The Watchdog (Automatic Recovery)

Inside the deployment folder, you will find WATCHDOG.bat. When launched, it starts the VMS and monitors the process. If PizzaVMS.exe closes or crashes, the Watchdog waits 5 seconds and launches it again automatically.

2. Silent Mode (Vexel "Ghost" Tech)

Standard batch files show a black command prompt window. To keep the desktop clean, use START_SILENT.vbs. This script launches the Watchdog in the background — no windows, no clutter. The only indication it is running is the Pizza & Ice Cream Club Tray Icon.

💡 Pro Tip: Auto-Start on Boot
To make the AI start automatically when the computer is turned on:
  1. Press Win + R, type shell:startup and hit Enter.
  2. Create a Shortcut to START_SILENT.vbs in that folder.
  3. The AI will now launch silently every time Windows boots up.

06 ❓ Frequently Asked Questions

The browser says "This site can't be reached" after I change the port

This usually means the existing PizzaClub.exe is still listening on the old compiled port. Either open http://127.0.0.1:8000 (the original port), or rebuild the EXE using BUILD_NEW_EXE.bat so the new port is baked in.

Can I run two instances on the same PC?

Yes. Copy the entire application folder to a second location, give each its own config.json with a different "app_port", and both servers will run independently. Useful for monitoring two branches from one machine.

The video is smoother but counting accuracy looks the same — is that right?

Yes. The YOLO model still runs at imgsz=480 which is completely unrelated to the display resolution. More frames per second means more detection opportunities for fast-moving objects — counting accuracy can only improve, never decrease.

The remote ngrok link still showing the old port

Close ngrok and re-run START_REMOTE_ACCESS.bat. The batch script now reads "app_port" from config.json each time it runs, so a fresh start will pick up the new value automatically.

How do I revert to port 8000?

Open config.json and change "app_port" back to 8000. If you have the old compiled EXE, it will already be listening on 8000 — no rebuild needed.

Where are my logs and audit records stored?

pizza_logs.json and audit_log.json are always stored next to the .exe in the same folder. They are never inside _internal/ and are safe across rebuilds — simply copy them to the new deployment folder.
